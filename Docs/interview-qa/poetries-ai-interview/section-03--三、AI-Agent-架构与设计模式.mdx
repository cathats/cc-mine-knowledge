<QAHtmlItem question={`43. Plan-and-Execute Agent 和 ReAct 的区别？`} html={`<ul><li><strong>ReAct</strong>：边想边做。优点灵活，缺点中间一旦判断错容易跑偏；</li> <li><strong>Plan-and-Execute</strong>：<strong>先一次性规划出所有步骤</strong>，再按计划执行，必要时回头改计划。优点路径稳定，缺点对长程任务一次规划容易遗漏。</li></ul>
<p>工程上常用<strong>混合</strong>：先 plan 出粗粒度大步骤，每个大步骤内部跑 ReAct 小循环。</p>`} />

<QAHtmlItem question={`44. Agent Loop 在工程里有哪些常见坑？`} html={`<p>跑过的人都被坑过：</p>
<ol><li><strong>死循环 / 振荡</strong>：模型反复调同一工具或在两个工具之间来回横跳；</li> <li><strong>Token 烧得飞快</strong>：每轮都把全量上下文带过去，几轮翻好几倍；</li> <li><strong>错误放大</strong>：第一步观察就理解错了，后面跟着错的方向越走越远；</li> <li><strong>越权 / 安全风险</strong>：没限制的话模型会调危险工具（删库、发邮件、<code>rm -rf</code>）；</li> <li><strong>不会停</strong>：自我反思没有终止条件，永远在"再确认一下"。</li></ol>
<p>兜底动作（必做）：</p>
<ul><li>设硬上限：<strong>max_steps、max_tokens、max_duration</strong>；</li> <li><strong>重复检测</strong>：同一工具同一参数连续 N 次直接停；</li> <li><strong>工具白名单 + 危险操作二次确认</strong>；</li> <li><strong>全链路日志</strong>：每步 prompt / tool_call / result 都落盘，事后能回放；</li> <li><strong>上下文压缩</strong>：旧轮次定期摘要。</li></ul>`} />

<QAHtmlItem question={`45. Agent 怎么自己判断"任务完成了"？`} html={`<p>几种常见判定，按工程靠谱程度：</p>
<ol><li><strong>显式工具返回 <code>final_answer</code></strong>：模型主动调"完成"工具；</li> <li><strong>没有 tool_calls 的回答</strong>：模型直接给文本输出（最常见）；</li> <li><strong>触发停止词 / 标签</strong>：比如输出 <code>&lt;done&gt;</code> 后中断；</li> <li><strong>外部判定</strong>：跑一个评测 LLM 判断"够不够"；</li> <li><strong>轮次上限</strong>：兜底，不能省。</li></ol>
<p>不要只靠"模型自己判断"。</p>`} />

<QAHtmlItem question={`46. 单 Agent vs 多 Agent 怎么选？`} html={`<p>复杂 Agent 产品基本是<strong>多 Agent 架构</strong>。原因：</p>
<ol><li><strong>决策准确率更高</strong>：每个子 Agent 只看自己需要的 prompt，没有无关信息干扰；</li> <li><strong>Token 更省</strong>：不是每次都把所有工具描述塞进去，调用变多但单次更便宜；</li> <li><strong>可并行</strong>：主 Agent 派活，子 Agent 并发处理；</li> <li><strong>多角色互相纠错</strong>：A 写代码 B 评审 C 验证，比单 Agent 自反思靠谱。</li></ol>
<p>什么时候<strong>不要</strong>多 Agent？</p>
<ul><li>任务简单、步骤少、用户期待秒级响应；</li> <li>团队还没把单 Agent 跑稳就上多 Agent，反而更难调试。</li></ul>
<blockquote><p>拿 Cursor / Claude Code 类比：早期是单 Agent，现在内部分了 plan、edit、verify、search 多个角色。</p></blockquote>`} />

<div class="language- extra-class"><pre class="language-text"><code>1. 跑一遍任务 → 拿到 result_v1

<QAHtmlItem question={`51. Agent 工具描述写不好会怎样？`} html={`<p>模型理解工具靠的就是描述。描述写不好直接表现为：</p>
<ul><li>模型<strong>该调没调</strong>（不知道这个工具能解决）；</li> <li><strong>调错工具</strong>（两个工具描述太像）；</li> <li><strong>传错参数</strong>（schema 字段名 / 必填项 / 枚举值没写清）；</li> <li><strong>传冗余参数</strong>（描述太长，模型脑补出不存在的字段）。</li></ul>
<p>工具描述模板（亲测好用）：</p>
<div class="language- extra-class"><pre class="language-text"><code>name: search_user_orders
description: 根据用户 ID 查询该用户最近 30 天的订单列表，
  返回订单 ID、状态、金额。**仅用于查询，不能创建或修改订单**。
parameters:
  user_id (string, required): 用户的唯一 ID，类似 'U-12345'。
  limit (number, optional, default=10): 返回数量，最大 50。
</code></pre></div>
<p>经验：</p>
<ul><li><strong>明确"能做什么"和"不能做什么"</strong>；</li> <li>给一个<strong>典型用法</strong>例子（用 # example 字段）；</li> <li>字段写<strong>取值范围、单位、格式</strong>；</li> <li>工具数量超过 15 个就要分组 / 分层加载，否则模型选错率飙升。</li></ul>`} />

<QAHtmlItem question={`52. Agent 怎么调试？看哪些指标？`} html={`<p>最少四件套：</p>
<ol><li><strong>每步 prompt + tool_call + result 日志</strong>（落盘 + 可搜索）；</li> <li><strong>步数分布</strong>：均值、p95、超过 N 步的 trace；</li> <li><strong>Token 消耗分布</strong>：哪些工具调用 token 异常高；</li> <li><strong>失败模式聚合</strong>：把失败 trace 按错误类型聚类，看是 prompt 问题 / 工具问题 / 模型问题。</li></ol>
<p>工具推荐：<strong>LangSmith</strong>（最常见）、Helicone、Phoenix、Langfuse。</p>`} />

<QAHtmlItem question={`53. Human-in-the-loop（HIL）是什么？什么时候必加？`} html={`<p>让用户在 Agent 关键步骤<strong>介入决策</strong>。</p>
<p>必加的场景：</p>
<ul><li><strong>不可逆操作</strong>：发邮件、付款、删数据；</li> <li><strong>金额超阈值</strong>：比如退款 &gt; 500；</li> <li><strong>跨权限操作</strong>：访问别的部门数据；</li> <li><strong>法律 / 合规风险点</strong>。</li></ul>
<p>实现方式：Agent 跑到关键节点时，<strong>返回一个 pending 状态</strong>给前端，前端展示"是否继续"按钮，用户确认后再继续。状态保存到 DB，断线能恢复。</p>`} />

<QAHtmlItem question={`54. Agent 的成本怎么优化？`} html={`<p>一套组合拳：</p>
<ol><li><strong>小模型分流</strong>：简单分类 / 路由用 7B 小模型，复杂任务才给大模型；</li> <li><strong>Prompt Caching</strong>：长 system + 工具描述全部缓存；</li> <li><strong>工具结果裁剪</strong>：返回前砍掉不必要字段，别让模型读完整 SQL 输出；</li> <li><strong>历史摘要</strong>：旧轮次定期压成几句话；</li> <li><strong>并行 vs 串行</strong>：能并行的工具调用一定要并行（多数 SDK 支持）；</li> <li><strong>缓存层</strong>：常见问答先查缓存 / 向量库再决定要不要调模型。</li></ol>`} />

<QAHtmlItem question={`55. LLM-as-Judge 是什么？`} html={`<p>用一个 LLM 评估另一个 LLM 的输出。</p>
<p>典型场景：</p>
<ul><li>跑 prompt 改进 A/B，让"裁判模型"对比两个回答打分；</li> <li>Agent 跑完后，让裁判模型判断"是否完成任务"；</li> <li>给数据集打偏好标签，省人工。</li></ul>
<p>注意：</p>
<ul><li>裁判模型要<strong>比被评估模型更强</strong>（GPT-4 当 Claude 4 的裁判没问题，反过来不行）；</li> <li><strong>明确评估维度</strong>（准确性 / 相关性 / 格式 / 礼貌），别让它笼统打个分；</li> <li>跑多个裁判取平均，<strong>减少裁判模型本身的偏差</strong>。</li></ul>`} />

<QAHtmlItem question={`56. Agent 的延迟（Latency）怎么优化？`} html={`<p>最影响体感的几个动作：</p>
<ol><li><strong>streaming</strong>：首 token 时间从几秒降到 1 秒内；</li> <li><strong>并行工具调用</strong>：多个工具同时跑而不是串；</li> <li><strong>小模型预筛 + 大模型精排</strong>；</li> <li><strong>预热 / Keep-alive</strong>：减少连接建立开销；</li> <li><strong>Prompt Caching 命中率</strong>；</li> <li><strong>边缘部署 / CDN</strong>：让首跳 RTT 更短。</li></ol>`} />

<QAHtmlItem question={`57. Agent 怎么测？写得了单元测试吗？`} html={`<p>写得了，但要换思路：</p>
<ul><li><strong>确定性部分</strong>：工具函数、parser、router 当普通函数测；</li> <li><strong>LLM 调用部分</strong>：mock 掉，或者用<strong>录制 / 回放</strong>（vcr 模式）；</li> <li><strong>整体端到端</strong>：写一组业务测试用例，跑全量，<strong>用 LLM-as-Judge 判通过</strong>，统计通过率作为回归指标。</li></ul>
<p>不要追求 100% 通过，AI 系统天然有不确定性，<strong>关注趋势</strong>：每次改完通过率涨还是跌。</p>`} />

<QAHtmlItem question={`58. AutoGPT、BabyAGI、Devin 这类"全自主 Agent"为什么不实用？`} html={`<p>理论很美，工程上一堆问题：</p>
<ul><li><strong>目标偏移</strong>：开放性任务模型容易越走越远；</li> <li><strong>token 爆炸</strong>：跑几小时一两千美元下去了；</li> <li><strong>错误传播</strong>：早期错误后期放大；</li> <li><strong>没有 HIL 节点</strong>，跑错也没人能拦；</li> <li><strong>可观测差</strong>：跑完不知道为啥成 / 为啥败。</li></ul>
<p>教训：<strong>生产用的 Agent 都是"半自主 + 有边界 + 有 HIL"</strong>，不是完全放飞。</p>`} />

<QAHtmlItem question={`59. 怎么把现有产品改造成 Agent？`} html={`<p>务实步骤：</p>
<ol><li><strong>找一个高价值的 workflow</strong>（比如客服处理退款），不要一上来全产品 Agent 化；</li> <li><strong>梳理这个 workflow 的所有手动步骤</strong>：每一步用了哪个内部接口；</li> <li>把这些接口<strong>包成 Tool</strong>（写好 schema + 描述）；</li> <li>写一个<strong>严格的 system prompt</strong>，限制 Agent 只能在这个 workflow 内做事；</li> <li><strong>HIL 卡关键节点</strong>（涉及钱、涉及不可逆）；</li> <li><strong>小流量灰度</strong>（5% → 20% → 50% → 100%），看转化和投诉。</li></ol>
<p>不要"先做个 Agent 框架再说"。</p>`} />

<QAHtmlItem question={`60. 一个 Agent 的"思考预算"应该怎么设？`} html={`<p>三个维度：</p>
<ul><li><strong>最大步数</strong>（max_steps）：8 - 20 是常见范围，超过基本是死循环；</li> <li><strong>最大 token</strong>（max_total_tokens）：按业务任务复杂度估，比如 50k；</li> <li><strong>最大耗时</strong>（max_duration）：用户能忍受的上限，通常 60-120 秒。</li></ul>
<p>任何一个达到上限就<strong>优雅终止</strong>：返回部分结果 + 说明"任务未完成"，<strong>不要让用户等永远</strong>。</p>`} />

<QAHtmlItem question={`93. 工具描述写得好不好直接决定调用准不准？`} html={`<p>模型理解工具只靠你写的 description + parameters schema。写得不好会出三类问题：</p>
<ol><li><strong>该调没调</strong>：描述模糊，模型没意识到这个工具能解决；</li> <li><strong>调错工具</strong>：两个工具描述相似，模型抓阄；</li> <li><strong>参数错</strong>：字段名 / 必填 / 枚举值 / 单位没说清。</li></ol>
<p>实战经验：</p>
<ul><li>描述写"<strong>能做什么</strong>"+ "<strong>不能做什么</strong>" + "<strong>典型用法</strong>"；</li> <li>参数字段加 <strong>取值范围、单位、格式、示例</strong>；</li> <li>必填的字段 description 里再强调一遍 "Required";</li> <li>工具数量超过 15 个就分组加载，<strong>不要全部塞 system</strong>。</li></ul>`} />

<QAHtmlItem question={`97. 工具结果太大怎么办？`} html={`<p>工具返回 50k token 直接塞回 LLM 就炸了。常见做法：</p>

<QAHtmlItem question={`99. MCP 是什么？为什么需要它？`} html={`<p>MCP = <strong>Model Context Protocol</strong>（模型上下文协议，Anthropic 提出，OpenAI / Google 等厂商陆续支持）。</p>
<p>要解决的问题：<strong>Function Calling 没有跨厂商 / 跨进程 / 跨语言的标准协议</strong>。</p>
<p>每个厂商自己定义 Tool Calling 的格式，工具是用 Python 写的还是 Node 写的都得重新接一遍。MCP 把这件事<strong>协议化</strong>：</p>
<ul><li>工具方实现 MCP Server（用什么语言无所谓，按协议跑就行）；</li> <li>客户端（Claude Code / Cursor / 自研 Agent）通过 MCP 协议连这些 Server；</li> <li>协议规定了 list_tools、call_tool、list_resources、list_prompts 等标准接口（底层基于 JSON-RPC）。</li></ul>
<p>效果：<strong>写一次工具，所有支持 MCP 的客户端都能用</strong>。</p>`} />

<QAHtmlItem question={`100. MCP 的三层核心结构（Server / Client / Bridge）`} html={`<p>很多面试官会追问 MCP 的结构，按三角色记：</p>
<ul><li><strong>Server（工具服务端）</strong>：提供一组可用的工具 / 资源 / Prompt（如文件读写、数据库、HTTP 请求、图像生成）；</li> <li><strong>Client（模型 / IDE / Agent）</strong>：通过协议访问 Server，把工具能力暴露给 LLM；</li> <li><strong>Bridge（中间层）</strong>：负责协议转发、权限校验、上下文同步、日志审计。Claude Code / Cursor 自身就充当 Bridge 角色。</li></ul>
<p>调用链路：</p>
<div class="language- extra-class"><pre class="language-text"><code>LLM → Function Calling → MCP Client → (Bridge) → MCP Server → 真实外部资源
</code></pre></div>
<p>工程上：Server 由工具方独立维护，Client 由 Agent 框架实现，Bridge 决定了"哪些工具能被哪个用户哪个会话调用"，是企业级权限和审计的关键点。</p>`} />

<QAHtmlItem question={`101. MCP 和 Function Calling 是替代关系吗？`} html={`<p>不是。</p>
<ul><li>Function Calling 是<strong>模型理解工具的方式</strong>（schema → JSON 调用）；</li> <li>MCP 是<strong>工具进程间通信的协议</strong>（stdio / HTTP / SSE）。</li></ul>
<p>模型还是用 Function Calling 调工具，<strong>只是工具的实现可以放在另一个 MCP Server 里</strong>，跨语言、跨进程、跨机器。</p>
<p>类比 HTTP 和 RESTful：HTTP 是协议，RESTful 是风格，MCP 让"工具的封装"也变得跨生态可复用。</p>`} />

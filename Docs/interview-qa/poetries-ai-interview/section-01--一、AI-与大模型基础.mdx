<QAHtmlItem question={`1. 什么是大语言模型（LLM）？它和过去的 NLP 模型本质区别是什么？`} html={`<p><strong>考察点</strong>：能否用一句话讲清，并知道它不是「智能」而是「概率」。</p>
<p>LLM 本质上是一个<strong>只会做一件事的概率模型</strong>：给一段上文，预测下一个 token 最可能是什么。把这个动作循环执行，就能输出一句话、一段代码、一篇文章。</p>
<p>它和过去 NLP 的区别有两点最关键：</p>
<ul><li><strong>不再针对单一任务训练</strong>。过去要做分类训分类，做翻译训翻译；现在一个模型靠 prompt 切换几十种任务。</li> <li><strong>规模带来质变</strong>。参数量到一定规模后，会<strong>涌现</strong>出推理、代码、跨语言能力，这是小模型里不存在的。</li></ul>
<blockquote><p>一句话总结：本质是「自回归概率分布」+「大力出奇迹」，不要把它当成会思考的人。</p></blockquote>`} />

<QAHtmlItem question={`3. Token 是什么？为什么按 token 计费而不是字符？`} html={`<p>Token 是<strong>模型实际处理的最小单元</strong>，由 tokenizer 把文本切成一串整数 id。模型从头到尾看到的都是 token id，不是字符。</p>
<p>按 token 计费的真正原因：<strong>模型的算力消耗和 token 数线性相关</strong>，和字符数不是。比如同一句话英文 4 token、中文可能 8 token，按字符算就不公平了。</p>
<p>常用换算（够你在面试里给个数）：</p>
<ul><li>英文：1 token ≈ 4 字符 ≈ 0.75 个单词</li> <li>中文：1 token ≈ 1.5 个汉字（不同模型差别不小）</li> <li>代码：和英文接近，但符号多的语言会更碎</li></ul>
<p>实战经验：<strong>估成本时给 1.3 倍 buffer</strong>，因为输入和输出都会算，且 system prompt 在多轮里会反复带。</p>`} />

<QAHtmlItem question={`4. 上下文窗口（Context Window）是什么？越大越好吗？`} html={`<p>窗口 = 模型一次能塞进去的 token 总额 = <code>system + 历史对话 + 当前输入 + 留给输出的预算</code>。超过就要截断、报错或者换策略。</p>
<p>不是越大越好，几个反常识的点：</p>
<ul><li><strong>大窗口 ≠ 注意力均匀</strong>。模型对开头和结尾的内容关注更高，<strong>中间段容易被忽略</strong>，叫「Lost in the Middle」。</li> <li><strong>成本和延迟一起涨</strong>。1M token 上下文每次调用都极贵，而且首 token 时间会肉眼可见地变慢。</li> <li><strong>大窗口不能完全替代 RAG</strong>。把 50 篇文档全塞进去，效果通常不如 RAG 检索出最相关的 5 段塞进去。</li></ul>
<p>工程上反而要<strong>主动控制</strong>：滑动窗口、摘要压缩、向量检索拼接才是常态。</p>`} />

<QAHtmlItem question={`5. Temperature、top_p、top_k、seed 这几个参数怎么调？`} html={`<p>记住一句话：<strong>调一个就够，别全都调</strong>。最常用的是 <code>temperature</code>。</p>
<table><thead><tr><th>参数</th> <th>控制什么</th> <th>代码 / 抽取场景</th> <th>创意 / 闲聊</th></tr></thead> <tbody><tr><td>temperature</td> <td>整体随机性</td> <td>0 - 0.3</td> <td>0.7 - 1.0</td></tr> <tr><td>top_p</td> <td>截断到累计概率 p</td> <td>默认 0.9 不动</td> <td>同上</td></tr> <tr><td>top_k</td> <td>截断到前 k 个候选</td> <td>一般不调</td> <td>一般不调</td></tr> <tr><td>seed</td> <td>复现性（部分模型支持）</td> <td>固定 seed + temperature=0</td> <td>—</td></tr></tbody></table>
<p>要让结果<strong>完全可复现</strong>：固定 seed + temperature=0 + top_p=1，且不要打开 streaming（部分 SDK streaming 有微差）。</p>`} />

<QAHtmlItem question={`6. Embedding 是什么？和 LLM 的关系？`} html={`<p>Embedding 模型是<strong>专门把一段文本压成一个固定长度向量</strong>（比如 1024 / 1536 / 3072 维）的模型，向量距离反映语义相似度。</p>
<p>它和 LLM 是<strong>两个独立模型</strong>：</p>
<ul><li>LLM：处理理解 + 生成</li> <li>Embedding 模型：只做表示，不生成</li></ul>
<p>典型用途：语义搜索、RAG 检索、聚类、推荐、查重、相似工单合并。</p>
<p>面试常被追问的细节：</p>
<ul><li><strong>维度越高 ≠ 效果越好</strong>，主流 1024 / 1536 已经够用，维度高会让向量库存储和检索都变贵；</li> <li><strong>同一份语料要用同一个 embedding 模型</strong>，换模型必须全量重建索引；</li> <li><strong>中文场景优先选支持中文的模型</strong>（bge-m3、Qwen-embedding、jina-zh），用英文 embedding 在中文上效果会折损。</li></ul>`} />

<QAHtmlItem question={`7. 余弦相似度 vs 欧氏距离怎么选？`} html={`<p>向量检索里 99% 用<strong>余弦相似度</strong>：只看方向不看长度，对文本归一化更友好。</p>
<p>欧氏距离会受向量模长影响，多用在「位置 / 物理量 / 图像特征」这种长度本身有意义的场景。</p>
<p>实战记一句话：<strong>embedding 都归一化之后，余弦相似度和欧氏距离结果基本等价</strong>（差一个常数变换），所以选哪个看库支持。</p>`} />

<QAHtmlItem question={`8. 模型「幻觉」（Hallucination）的根因是什么？`} html={`<p>不是 bug，是<strong>架构决定的特性</strong>。</p>
<p>LLM 在做"下一个 token 概率最大化"，并不会去查事实。当训练里没见过对应内容，它会<strong>按统计相关性继续编</strong>——表达通顺、事实瞎说。</p>
<p>减少幻觉是组合拳，没有单点药方：</p>
<ol><li><strong>接 RAG</strong>，让答案有据可查（这是最有效的一招）；</li> <li>在 system prompt 显式要求 <strong>"不知道就说不知道，不要编"</strong>，并给一个示范；</li> <li>强约束<strong>输出格式</strong>，比如必须 JSON + 必须带引用片段编号；</li> <li>用代码或第二个 LLM 做<strong>后置校验</strong>（金额、日期、URL 这类容易瞎编的字段重点查）；</li> <li><code>temperature</code> 拉到 0 - 0.3；</li> <li>实在不行才上<strong>微调</strong>（成本最高）。</li></ol>`} />

<QAHtmlItem question={`9. 自回归（Decoder-only）和 Encoder-Decoder 区别？`} html={`<p>自回归：从左到右一个 token 一个 token 生成，代表 GPT / Claude / Llama / Qwen，<strong>现在主流 LLM 都是这个架构</strong>。</p>
<p>Encoder-Decoder：先把整个输入编码成中间表示，再解码出来，代表 T5 / BART，<strong>输入输出有明确对应</strong>的任务（翻译、摘要、纠错）历史上效果好。</p>
<p>为什么现在都偏向 Decoder-only？工程上更简单：一个模型搞定所有任务，只看 prompt。</p>`} />

<QAHtmlItem question={`10. KV Cache 是什么？为什么能加速生成？`} html={`<p>自回归生成时，<strong>每个新 token 都依赖前面所有 token 的 Key/Value</strong>。如果每生成一个 token 都重算一遍前面的 KV，复杂度 O(n²) 极慢。</p>
<p>KV Cache 的本质：<strong>把已经算过的 K、V 存下来，新 token 只追加计算</strong>，复杂度降到接近 O(n)。</p>
<p>附带几个延伸：</p>
<ul><li><strong>Prompt Caching</strong>（Anthropic / OpenAI 都支持）就是把不变的 system prompt 部分 KV 复用，对长 system + 多轮对话的场景能省很多钱；</li> <li>KV Cache 占显存，所以推理服务的并发上限就被显存卡住。</li></ul>`} />

<QAHtmlItem question={`12. System Prompt 是干什么的？应该放些什么？`} html={`<p>System Prompt 是<strong>整段对话的"底色"</strong>，模型每轮回答前都会先读它。</p>
<p>四要素，最好按顺序写：</p>
<ol><li><strong>角色定位</strong>：你是谁、你的专业是什么；</li> <li><strong>任务边界</strong>：能做什么、不能做什么；</li> <li><strong>风格 / 格式</strong>：语气、长度、是否带代码、是否带引用；</li> <li><strong>兜底规则</strong>：不知道就说不知道、敏感问题怎么处理、什么时候让用户兜底。</li></ol>
<p>实战经验：</p>
<ul><li>把"重要约束"放在 system 的<strong>开头和结尾</strong>，中间放细节，因为中间最容易被模型忽略；</li> <li>不要在 system 里塞知识，那是 RAG 的工作；</li> <li>system 不变的部分尽量复用，触发 Prompt Caching。</li></ul>`} />

<QAHtmlItem question={`13. 多模态模型是什么？前端怎么用？`} html={`<p>多模态 = 能处理文本之外的东西。常见组合：</p>
<ul><li>文本 + 图（GPT-4o、Claude 4 Vision、Gemini、Qwen-VL）</li> <li>文本 + 音频（Whisper STT、Realtime API）</li> <li>文本 + 视频（Gemini、Sora 类的生成）</li></ul>
<p>前端场景：</p>
<ul><li>上传截图问"这块 UI 该用什么组件"</li> <li>把设计稿喂给模型生成 HTML / Tailwind</li> <li>录音转文字、做语音输入</li> <li>把后端长报表截图扔上去做摘要</li></ul>
<p>注意 token 计费：一张图常常等于 1k - 2k token，比文本贵很多，要做大小压缩和 base64 vs URL 的取舍。</p>`} />

<QAHtmlItem question={`14. 模型的「知识截止时间」（Knowledge Cutoff）是什么？`} html={`<p>模型训练数据冻结的日期。过了这个日期发生的事它<strong>全不知道</strong>（昨天发布的 Vue 新版本、上周的政策、今天的股价）。</p>
<p>工程上三种应对：</p>
<ol><li><strong>RAG / 联网检索</strong>补最新信息；</li> <li>system prompt 显式告诉它<strong>今天是几月几号</strong>，避免它瞎猜；</li> <li>让它<strong>不确定就说不确定</strong>，不要装懂。</li></ol>`} />

<QAHtmlItem question={`15. RLHF、DPO、Constitutional AI 三个是什么？`} html={`<p>都是<strong>对齐</strong>（Alignment）的方法，让预训练完只会"猜 token"的模型变得"会听人话"。</p>
<ul><li><strong>RLHF</strong>：让人给模型多个回答打分，训一个奖励模型，再用 PPO 强化学习。流程长、训练贵；</li> <li><strong>DPO / ORPO</strong>：跳过奖励模型，直接用「选中 / 拒绝」对来做监督学习。简单、稳定，<strong>现在主流团队偏好 DPO</strong>；</li> <li><strong>Constitutional AI</strong>（Anthropic 提的）：用一组人写的「宪法原则」让模型自己批改自己的输出，再用改后的数据再训一遍，<strong>减少人工标注</strong>。</li></ul>`} />

<QAHtmlItem question={`16. 推理（Inference）和训练（Training）的成本结构差异？`} html={`<ul><li>训练：<strong>一次性、超贵</strong>。GPT 级模型一次训练上千万到上亿美元；</li> <li>推理：<strong>单次便宜，累计巨贵</strong>。生产环境跑半年，推理成本会远远超过训练。</li></ul>
<p>前端调 API 付的是推理成本，所以<strong>省 token 才是日常的钱</strong>：Prompt Caching、压缩历史、控制 max_tokens、用小模型分流、复用 Embedding 索引。</p>`} />

<QAHtmlItem question={`17. 涌现能力（Emergent Ability）是真的吗？`} html={`<p>是真的，但也被一部分论文质疑过（说是因为评估指标不连续造成的错觉）。</p>
<p>工程上重要的不是争论它真假，而是<strong>记住一个直觉</strong>：当某个任务在 7B 上做不了，先想想 70B 能不能做了再说，不要立刻去微调。</p>`} />

<QAHtmlItem question={`18. MoE（专家混合）和稠密模型的区别？`} html={`<p>稠密模型：每次推理所有参数都激活。</p>
<p>MoE：模型里有 N 个「专家」子网络，每个 token 只路由到 K 个专家（比如 8 选 2），<strong>参数总量大但激活量小</strong>，所以 700B 的 MoE 推理速度可以接近 70B 稠密模型。</p>
<p>代表：Mixtral、DeepSeek-V3、GPT-4 据传也是。</p>
